1) HTML Basics

HTML (HyperText Markup Language) is a markup language used to represent the structure of web documents. Elements (tags) such as headings, paragraphs, lists, images and forms express the meaning and organization of content. HTML documents should be valid and well-structured to ensure consistent rendering across browsers and better compatibility with search engines and assistive technologies. The separation of concerns principle â€” structure in HTML, presentation in CSS, behavior in JavaScript â€” helps keep code maintainable and performant.

Notes & Best Practices
  • Always include <!doctype html> to opt into standards mode.
  • Set the document language via lang on the <html> element.
  • Place critical metadata (charset, viewport, title) in the head to avoid rendering issues.
  • Use semantic HTML to improve accessibility and SEO.

2) Text & Formatting

Text elements define document hierarchy and semantic meaning. Headings (<h1>â€“<h6>) express structure and outline; paragraphs (<p>) encapsulate blocks of prose. Inline elements (<strong>, <em>, <mark>, <abbr>) convey emphasis, importance, or abbreviations. Lists (<ul>, <ol>, <dl>) group related items. For preformatted text and code samples, use <pre> and <code> to preserve whitespace and semantically identify code. Proper use of text semantics improves machine understanding and assistive technology navigation.

Notes & Best Practices
  • Use headings to create a logical document outline â€” one <h1> per page representing the main topic.
  • Avoid using headings for visual styling; use CSS instead.
  • Wrap code in <pre><code> and consider syntax highlighting libraries for readability.

3) Links & Navigation

Anchors (<a>) create hyperlinks connecting resources on the web. Attributes like href (destination), target (how to open), and rel (relationship, security) control behavior. Descriptive link text helps users and assistive technologies. Navigation sections typically use <nav> and often present links in lists for predictable semantics. When opening external links in new tabs, include rel="noopener noreferrer" with target="_blank" to prevent access to the opening window and to improve performance/security.

Navigation and example links
Notes & Best Practices
  • Use meaningful link text that describes the destination (avoid "click here").
  • For accessibility, ensure links are keyboard-focusable and have visible focus styles via CSS.
  • Use rel="noopener noreferrer" with target="_blank".

4) Images & Multimedia

Images, audio, and video enhance content but must be used with care for performance and accessibility. Always provide descriptive alt text for images â€” if the image is purely decorative, use an empty alt (alt=""). For responsive delivery, supply variants with srcset and sizes or use the <picture> element for art direction. Video and audio should include controls and captions. Lazy-loading media and choosing modern formats (WebP/AVIF) improves performance.

Figure with caption and placeholder video
Notes & Best Practices
  • Include practical alt text that conveys the content purpose; use alt="" for purely decorative images.
  • Use responsive techniques (srcset/picture) to serve appropriately sized images to devices.
  • Provide captions/transcripts for video/audio to meet accessibility and legal requirements.

5) Tables

Tables are appropriate for displaying tabular data: rows and columns that represent records and fields. Use semantic table structure: <caption> for summary, <thead>, <tbody>, and <tfoot> to partition content. Use <th> for headers and include scope or headers where needed to clarify relationships for assistive tech. Do not use tables for visual layout; use CSS layout systems instead.

Schedule table
Notes & Best Practices
  • Use <caption> to provide a short description of the table's purpose.
  • Use scope="col" or scope="row" on <th> to help screen readers map headers to cells.
  • Prefer semantic HTML tables for data; for layout, use CSS grid/flexbox.

6) Forms & Input

Forms capture user-provided data. Use appropriate input types (email, tel, number, date, etc.) to improve validation and mobile input keyboards. Associate each input with a <label> for accessibility. Use fieldset and legend to group related controls. While client-side validation improves UX, always validate and sanitize data on the server. Provide clear error messages and maintain sensible focus behavior when errors occur.

Contact form
Notes & Best Practices
  • Always include a name attribute on inputs to ensure data is submitted to the server.
  • Use semantic grouping for accessibility (e.g., radio/checkbox groups inside fieldset).
  • Validate on both client and server; never trust client-side validation alone.

7) Semantic HTML

Semantic HTML uses elements that convey meaning about the content they contain: <nav> for navigation, <main> for primary content, <article> for standalone items, <section> for thematically grouped content, and so on. Semantic markup makes content more discoverable and accessible. Search engines and assistive technologies rely on these semantic signals to interpret page structure.

Semantic layout
Notes & Best Practices
  • Prefer semantic elements to generic <div> where the element's meaning is clear.
  • Landmark elements (header, nav, main, footer) assist screen reader navigation.
  • Use ARIA roles only when native elements do not provide the necessary semantics.

8) Attributes

Attributes add metadata or behavior to elements. Common global attributes include id, class, title, and style. Custom data attributes (data-*) store information for scripts, and ARIA attributes can be used to enhance accessibility. Use classes for styling and JavaScript hooks, keep id values unique, and avoid overusing inline styles for maintainability.

Attributes and data-* usage
Notes & Best Practices
  • Use data-* attributes to store small amounts of application data on elements.
  • Prefer classes for styling and JS hooks rather than element-specific IDs where possible for reusability.
  • Be careful with inline styles â€” they make maintenance and theming harder.

9) Graphics (SVG & Canvas)

SVG (Scalable Vector Graphics) is an XML-based format for vector graphics that scales cleanly and supports accessibility, styling, and scripting. Because SVG elements are part of the DOM, they can be manipulated with CSS/JS. Canvas provides a pixel-oriented drawing surface controlled by JavaScript and is suitable for games and intensive visualizations. When using canvas, provide fallback or textual descriptions for accessibility.

Notes & Best Practices
  • Use SVG for icons, diagrams, and any graphics that must scale cleanly.
  • Canvas is powerful but less accessible â€” provide alternate content or ARIA labeling.
  • Consider file size and complexity â€” very complex SVGs may perform poorly if not optimized.

10) Head & Meta

The <head> contains metadata that affects rendering, indexing, and linking. Key elements include <title>, character encoding, viewport meta for responsive scaling, links to stylesheets, and preconnect/preload hints. Meta description and Open Graph metadata improve how content appears in search results and social shares. Keep head content concise to avoid render-blocking issues.

Head content example
Notes & Best Practices
  • Write concise, descriptive titles and meta descriptions for SEO and UX.
  • Use resource hints (preload, prefetch) carefully to optimize loading.
  • Avoid large style/script blocks in the head that block rendering; inline critical CSS only when necessary.

11) Accessibility

Accessibility ensures web content is usable by people with disabilities. Start with semantic HTML and provide meaningful text alternatives, clear focus states, keyboard navigability, and sufficient color contrast. Use label elements with form controls, apply ARIA roles only when native semantics are insufficient, and test with assistive technologies. Accessibility benefits all users and is increasingly required by law in many regions.

Accessible form input example
Notes & Best Practices
  • Prefer native HTML semantics; only use ARIA to enhance where necessary.
  • Test with keyboard-only navigation and a screen reader (NVDA, VoiceOver, or JAWS).
  • Provide skip links and ensure logical tab order for complex layouts.

12) Advanced HTML

Advanced HTML features include interactive elements and integrations: <datalist> for suggestions, contenteditable for inline editing, web components for encapsulated UI, and many browser APIs (Fetch, Web Storage, Service Workers) that expand what web apps can do offline and interactively. Apply progressive enhancement: content should be useful without JavaScript, and features layered on top as capabilities are detected. Consider security implications for user-generated content and apply proper sanitization and Content Security Policy (CSP).

Notes & Best Practices
  • Use progressive enhancement: ensure basic functionality works without JavaScript.
  • Sanitize and validate any user-submitted HTML (contenteditable output) before rendering on other pages.
  • Explore Service Workers for caching and offline support, but handle updates carefully.
